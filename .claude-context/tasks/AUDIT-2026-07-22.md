## TASK-027: Dashboard cache — phantom data
**Stav v TASK-QUEUE:** ceka
**Skutecny stav v kodu:** ⚠️ ZMENSENO ALE NEODSTRANENO
**Dukaz:**
- `dashboard/page.tsx:172` — `revalidate: 10` (snizeno z puvodniho 60 sekund na 10 sekund)
- Tag-based invalidace funguje — `revalidateTag("dashboard", "max")` je volano z:
  - deliveries (POST, DELETE, PUT)
  - products (PUT)
  - sales (POST, PUT)
  - orders (cancel, status changes)
  - variants/purge
  - comgate/callback
- `recentMovements` query (radek 111-119): `stockMovement.findMany({ orderBy: { createdAt: "desc" }, take: 6 })` — zobrazuje pohyby z tabulky stockMovement. Pokud se pohyby smazou (napr. pri purge varianty), cache se revaliduje.
- **Mozny problem:** Cache `unstable_cache` s `revalidate: 10` muze stale ukazovat phantom data po dobu az 10 sekund. Take `revalidateTag` nemusi fungovat spolehlive u all deploys (Vercel edge cache).
**Zaver:** Cache interval snizen na 10s a tag invalidace zapojena ve vsech dulezitych API routes. Problem muze pretrvavat kvuli edge cache behavior. Doporucuji snizit na revalidate: 0 nebo pouzit no-store pro dashboard.
